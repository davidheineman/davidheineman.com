The afternoon I wrote my own parser

I have been using the same regex for seven years. Last Tuesday it finally lost a fight with a piece of YAML, and I ended up down a very pleasant rabbit hole.

April 14, 2026 9 min read

I will tell you up front: the regex was fine. It had earned its keep. It survived two job changes, a rewrite from Python to Go1The Go rewrite was less of a rewrite and more of a slow-motion translation in which I kept the variable names and lost most of the dignity. It also coincided with my brief, ill-advised aI bought the mechanical keyboard. I am not proud. The keyboard is, however, very nice. mechanical-keyboard phase., and the brief period in 2021 when I thought I was going to build everything in Rust. Every time someone on the team asked "couldn't we just write a real parser for this?" I would nod politely and then not do that.

What finally broke it was a config file with a quoted string containing a colon2If you have ever written a YAML parser by hand, you already know that the colon is a load-bearing character with strong opinions and a small army of sub-rules.. Obvious in retrospect. Embarrassingly obvious. The regex confidently split the line in the wrong place and the rest of the afternoon was me yelling at a stack trace with no meaningful line numbers in it.

Just write the loop

The thing nobody tells you about hand-rolling a parser for a small language is that it is, genuinely, a loop and a switch. I keep forgetting this. I keep reaching for parser combinators or PEG grammars or nom rust, sitting down with tea, and coming back an hour later having produced a file full of types and no actual parser.

This time I decided to be boring. A single index, a single input string, a function per thing-I-want-to-parse. No crates. No structs named Context. No dyn Trait. Just: 

const Parser = struct {
    src: []const u8,
    i:   usize = 0,

    fn peek(p: *Parser) ?u8 {
        return if (p.i < p.src.len) p.src[p.i] else null;
    }

    fn eat(p: *Parser, c: u8) bool {
        if (p.peek() == c) { p.i += 1; return true; }
        return false;
    }
};

That is it. That is the whole thing. Everything else in the parser is some combination of peek, eat, and a while loop. Somehow, for about eight years, I had been convinced this was a thing you needed libraries for3To be fair to past-me: when your grammar grows teeth — operator precedence, lookahead, error recovery — a real library does start to earn its keep. The trick is being honest about whether you're there yet..

The surprisingly good news about writing a parser by hand is that you get to decide what counts as a valid input, and nobody can stop you.

Strings are the hard part, but only a little

Before we get there: the reason I stopped reaching for libraries is that the cost of a hand-rolled parser is almost always $O(n)$ in the input length, and the constant factor is a handful of instructions per byte. You can even write down an honest upper bound on the total work:

$$ T(n) \;=\; \sum_{i=1}^{n} \big( c_{\text{peek}} + c_{\text{eat}} \big) \;+\; \sum_{r \in R} c_r \cdot |r| \;\le\; \alpha\,n + \beta $$

where $R$ is the set of rules I actually invoked, $|r|$ is how many characters each one consumed, and $\alpha, \beta$ are small integers that depend on how much I care. The punchline: a real library has to assume $R$ is arbitrary, but I know mine — there are nine rules and two of them are whitespace.

Of course, the reason I always bailed out at "strings with escapes" is not that strings with escapes are hard. It is that I kept trying to write a general-purpose escape handler on the first attempt, which meant thinking about Unicode, which meant thinking about whether I needed a code-point iterator, which meant opening Wikipedia, and now it is 4pm.

The fix is to not do that. Write the dumbest possible version first:

fn parseString(p: *Parser) ![]const u8 {
    if (!p.eat('"')) return error.ExpectedString;
    const start = p.i;
    while (p.peek()) |c| {
        if (c == '"') break;
        if (c == '\\') p.i += 1; // skip the next char, worry later
        p.i += 1;
    }
    const end = p.i;
    if (!p.eat('"')) return error.UnterminatedString;
    return p.src[start..end];
}

Is this correct for every input? Absolutely not. Does it correctly parse the ninety-something percent of strings my actual config files contain, including the one with the colon? Yes. Will I upgrade it later when something actually breaks? Also yes. That is the deal you make with yourself to get past the starting line.

The part where you feel smug

I want to be honest: there is a specific flavor of smugness you feel about two hours into writing a small parser by hand, when your tests are passing and the code fits on one screen. It is not proportional to the difficulty of what you did. It is, however, free.

The moment, captured for posterity.

What I wish I'd known

  1. The loop-and-switch version is not a rough draft. It is, most of the time, the final version.
  2. Write the tokenizer and the parser in the same file. Split them up only when the file gets annoying.
  3. Error messages are ninety percent of the work. Budget accordingly.
  4. If you find yourself reaching for a library, ask: is my grammar actually that big, or am I just scared?

The file I ended up with is 212 lines, including tests and comments. It replaces a regex, a clever split call, and about forty lines of post-hoc validation. It runs faster. It has line numbers in its errors. I can read it4This last one shouldn't feel like a feature, but here we are..

I'm not going to tell you to always write your own parser. Sometimes serde is right there and you should use it and go home. But if you have been avoiding a thing for seven years because it feels big, it is worth at least checking whether the thing is actually big or whether it is just wearing a costume.
